# 项目结构约定

本文档用于约定北京交通大学就餐仿真系统的代码目录结构。分支开发只能解决协作流程问题，真正的模块化需要靠统一目录、模块边界和接口约定来保证。

## 1. 顶层目录

```text
BJTU-Dining-Simulation-System/
├─ backend/        Spring Boot 后端
├─ frontend/       Vue3 前端
├─ docs/           项目文档、接口规范、分支约定
├─ sql/            建表脚本和初始化数据
└─ README.md
```

## 2. 后端结构

后端按业务模块拆包，避免所有代码堆在统一的 `controller`、`service`、`mapper` 大包里。

```text
backend/src/main/java/com/bjtu/dining/
├─ common/             通用响应、异常、配置、分页、安全工具
├─ auth/               登录、注册、JWT
├─ user/               用户资料、用户偏好
├─ wallet/             余额、充值
├─ order/              点餐、订单事务
├─ rating/             评分、评论、平均分回写
├─ restaurant/         餐厅基础信息
├─ window/             餐厅窗口基础信息
├─ dish/               菜品基础信息、库存维护
├─ flow/               人流仿真、拥挤度、等待时间
└─ recommendation/     餐厅推荐、菜品推荐
```

每个业务模块内部建议继续分层：

```text
模块名/
├─ controller/
├─ service/
├─ mapper/
├─ entity/
├─ dto/
└─ vo/
```

示例：

```text
dish/
├─ controller/DishController.java
├─ service/DishService.java
├─ mapper/DishMapper.java
├─ entity/Dish.java
├─ dto/DishQueryDTO.java
└─ vo/DishVO.java
```

## 3. 前端结构

前端按接口、页面、组件拆分，页面中不要直接散落 Axios 请求。

```text
frontend/src/
├─ api/            按业务模块封装请求
├─ views/          页面级组件
├─ components/     可复用组件
├─ router/         路由配置
├─ stores/         Pinia 状态
└─ utils/          工具函数
```

建议的 API 文件：

```text
frontend/src/api/
├─ auth.js
├─ user.js
├─ wallet.js
├─ order.js
├─ rating.js
├─ restaurant.js
├─ window.js
├─ dish.js
├─ flow.js
└─ recommendation.js
```

建议的页面：

```text
frontend/src/views/
├─ Login.vue
├─ Home.vue
├─ RestaurantDetail.vue
├─ DishDetail.vue
├─ UserCenter.vue
└─ Recommendation.vue
```

建议的组件目录：

```text
frontend/src/components/
├─ common/
├─ restaurant/
└─ dish/
```

## 4. SQL 结构

数据库脚本按建表和种子数据分开管理。

```text
sql/
├─ schema/
│  ├─ 001_user_wallet_order_rating.sql
│  ├─ 002_restaurant_window_dish.sql
│  └─ 003_flow_recommendation.sql
└─ seed/
   ├─ restaurant_seed.sql
   └─ dish_seed.sql
```

成员 B 负责维护：

```text
sql/schema/002_restaurant_window_dish.sql
sql/seed/restaurant_seed.sql
sql/seed/dish_seed.sql
```

## 5. 成员模块边界

| 成员 | 主要后端目录 | 主要前端目录 |
| --- | --- | --- |
| 成员 A | `auth/`、`user/`、`wallet/`、`order/`、`rating/` | `api/auth.js`、`api/user.js`、`api/wallet.js`、`api/order.js`、`api/rating.js`、`views/Login.vue`、`views/UserCenter.vue` |
| 成员 B | `restaurant/`、`window/`、`dish/` | `api/restaurant.js`、`api/window.js`、`api/dish.js`、`views/RestaurantDetail.vue`、`views/DishDetail.vue`、`components/restaurant/`、`components/dish/` |
| 成员 C | `flow/`、`recommendation/` | `api/flow.js`、`api/recommendation.js`、`views/Recommendation.vue`、图表相关组件 |

## 6. 跨模块调用规则

1. 前端统一通过 `frontend/src/api/` 下的封装方法调用后端，不在页面里直接拼接口路径。
2. 后端模块之间优先通过 Service 调用，不要在别人的模块里直接乱写 Mapper 查询。
3. 成员 A 下单时可以调用成员 B 的菜品库存服务，但库存字段和扣减规则由成员 B 维护。
4. 成员 C 推荐时可以读取成员 B 的餐厅、窗口、菜品数据，但不要修改这些基础数据。
5. 公共能力放入 `common/`，但 `common/` 只放真正通用的响应、异常、配置和工具，不能放具体业务逻辑。

## 7. PR 检查规则

提交 Pull Request 时需要说明：

```text
本次改动：
- 做了哪些功能

影响范围：
- 改了哪些模块和目录

不涉及：
- 哪些其他成员模块没有修改
```

如果改到了其他成员负责的模块，需要提前在群里说明。
