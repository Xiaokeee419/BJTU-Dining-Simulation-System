# 北京交通大学就餐仿真系统接口规范

版本：v0.2

日期：2026/4/25

适用范围：成员 A、成员 B、成员 C 并行开发和联调验收

---

## 1. 接口设计目标

本接口规范服务于新版“校园就餐场景仿真与决策支持系统”。系统不再以注册、充值、下单等管理流程为主线，而是以仿真运行和推荐决策为核心。

核心调用链如下：

```text
C 配置用户画像和场景参数
-> C 调用 A 的仿真接口
-> A 返回全体餐厅和窗口的排队、等待、拥挤度、评估指标
-> C 调用 B 的推荐接口
-> B 基于 A 的仿真结果返回餐厅、窗口、菜品推荐和分流建议
-> C 展示仿真控制台、动态图表和推荐结果
```

本规范目标：

1. 保证三名成员可以并行开发。
2. 保证 A、B、C 模块边界清晰。
3. 保证前端可以用统一数据结构展示全体排队状况和推荐结果。
4. 保证最终演示可以稳定完成“配置参数、运行仿真、动态展示、推荐决策、场景对比”的流程。

---

## 2. 成员接口归属

| 成员 | 负责范围 | 接口归属 |
| --- | --- | --- |
| 成员 A | 仿真数据生成与效果评估 | `/api/v1/presets/**`、`/api/v1/parameters/**`、`/api/v1/simulations/**` |
| 成员 B | 分流策略与推荐算法 | `/api/v1/recommendations/**`、`/api/v1/strategies/**` |
| 成员 C | 仿真控制台与可视化展示 | 不单独提供后端接口，负责调用 A、B 接口并展示 |

说明：

1. A 负责生成仿真结果，不负责推荐排序。
2. B 负责基于 A 的仿真结果做推荐和策略比较，不负责原始仿真数据生成。
3. C 负责页面展示，不直接计算仿真结果和推荐算法。

## 3. 全局接口约定

### 3.1 基础路径

所有 REST 接口统一使用：

```text
/api/v1
```

示例：

```text
POST /api/v1/simulations/run
POST /api/v1/recommendations/generate
```

### 3.2 联调环境

本地联调默认地址：

```text
后端服务：http://localhost:8080
前端服务：http://localhost:5173
接口前缀：http://localhost:8080/api/v1
```

后端需要允许前端跨域访问：

```text
允许来源：http://localhost:5173
允许方法：GET、POST、PUT、DELETE、OPTIONS
允许请求头：Content-Type、Authorization
```

如果某个成员本地端口不同，需要在群里提前说明，并保证接口路径不变。

### 3.3 鉴权约定

MVP 阶段暂不要求登录鉴权，所有仿真与推荐接口默认公开可访问。

如果后续补充登录功能，前端统一在 Header 中加入：

```http
Authorization: Bearer <token>
```

当前版本不要求 C 在请求中传真实用户 ID，用户输入以“用户画像参数”为准。

### 3.4 请求格式

1. 请求和响应统一使用 JSON。
2. Header 使用 `Content-Type: application/json`。
3. 字段统一使用 `camelCase`。
4. 金额字段使用数字类型，保留两位小数。
5. 时间使用 ISO-8601 字符串，例如 `2026-04-25T12:30:00+08:00`。
6. ID 字段统一使用数字类型，例如 `runId`、`restaurantId`、`windowId`、`dishId`。
7. 可为空字段使用 `null`，不要使用空字符串表示空值。

### 3.5 统一响应结构

所有接口统一返回：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "traceId": "9bdb3c3c6c7b4f41",
  "timestamp": "2026-04-25T12:30:00+08:00"
}
```

失败响应：

```json
{
  "code": 40001,
  "message": "参数校验失败",
  "data": {
    "field": "virtualUserCount",
    "reason": "仿真人数必须在 1 到 5000 之间"
  },
  "traceId": "9bdb3c3c6c7b4f41",
  "timestamp": "2026-04-25T12:30:00+08:00"
}
```

实现要求：

1. 成功时 `code` 必须为 `0`。
2. 失败时 `code` 不得为 `0`，`message` 必须能说明错误原因。
3. `traceId` 可由后端生成；MVP 阶段可以用任意唯一字符串。
4. C 判断接口是否成功时，优先判断响应体中的 `code`。

### 3.6 HTTP 状态码

MVP 阶段业务接口成功统一返回 HTTP `200`，业务结果通过响应体 `code` 判断。

| HTTP 状态码 | 使用场景 |
| --- | --- |
| 200 | 请求成功 |
| 400 | 参数错误 |
| 404 | 资源不存在 |
| 409 | 当前状态不允许操作 |
| 500 | 服务端异常 |
| 503 | 仿真或推荐模块暂不可用 |

### 3.7 业务错误码

| code | 含义 |
| --- | --- |
| 0 | 成功 |
| 40000 | 请求格式错误 |
| 40001 | 参数校验失败 |
| 40002 | 枚举值非法 |
| 40003 | 场景参数非法 |
| 40400 | 资源不存在 |
| 40401 | 仿真运行记录不存在 |
| 40402 | 参数预设不存在 |
| 40900 | 资源状态冲突 |
| 40901 | 仿真尚未完成，无法生成推荐 |
| 50000 | 服务端异常 |
| 50300 | 仿真服务暂不可用 |
| 50301 | 推荐服务暂不可用 |

### 3.8 分页约定

如需分页，统一使用：

```text
page=1&pageSize=10
```

分页返回：

```json
{
  "list": [],
  "total": 100,
  "page": 1,
  "pageSize": 10
}
```

当前 MVP 的核心仿真结果可以一次性返回完整时间序列，不强制分页。

## 4. 枚举定义

### 4.1 用户类型 `userType`

| 值 | 含义 |
| --- | --- |
| STUDENT | 学生 |
| FACULTY | 教职工 |
| VISITOR | 访客 |
| HURRY | 赶时间用户 |
| BUDGET_SENSITIVE | 预算敏感用户 |

### 4.2 时间段 `mealPeriod`

| 值 | 含义 |
| --- | --- |
| BREAKFAST | 早餐 |
| LUNCH | 午餐 |
| DINNER | 晚餐 |

### 4.3 日期类型 `dayType`

| 值 | 含义 |
| --- | --- |
| WEEKDAY | 工作日 |
| WEEKEND | 周末 |

### 4.4 拥挤等级 `crowdLevel`

| 值 | 含义 |
| --- | --- |
| IDLE | 空闲 |
| NORMAL | 正常 |
| BUSY | 拥挤 |
| EXTREME | 极端拥挤 |

### 4.5 营业状态 `status`

| 值 | 含义 |
| --- | --- |
| OPEN | 营业中 |
| CLOSED | 已关闭 |

### 4.6 仿真运行状态 `runStatus`

| 值 | 含义 |
| --- | --- |
| FINISHED | 已完成 |
| FAILED | 失败 |

MVP 阶段仿真接口采用同步返回，调用成功后直接返回 `FINISHED`。

### 4.7 推荐目标类型 `targetType`

| 值 | 含义 |
| --- | --- |
| RESTAURANT | 餐厅 |
| WINDOW | 窗口 |
| DISH | 菜品 |

### 4.8 拥挤度判定规则

为保证 A 生成结果、B 计算推荐、C 展示颜色一致，MVP 阶段统一使用以下判定规则。

餐厅拥挤度按 `currentCount / capacity` 计算：

| 比例范围 | crowdLevel |
| --- | --- |
| `[0, 0.4)` | IDLE |
| `[0.4, 0.7)` | NORMAL |
| `[0.7, 0.9)` | BUSY |
| `[0.9, +∞)` | EXTREME |

窗口拥挤度优先按 `waitMinutes` 计算：

| 等待时间 | crowdLevel |
| --- | --- |
| `< 5` 分钟 | IDLE |
| `5-10` 分钟 | NORMAL |
| `10-20` 分钟 | BUSY |
| `>= 20` 分钟 | EXTREME |

如果窗口状态为 `CLOSED`，`crowdLevel` 固定返回 `IDLE`，`queueLength`、`servingCount`、`waitMinutes` 固定返回 `0`。

## 5. 核心数据对象

### 5.1 用户画像 `UserProfileConfig`

```json
{
  "profileId": "student-spicy-budget",
  "name": "偏辣预算型学生",
  "userType": "STUDENT",
  "tasteTags": ["偏辣", "米饭"],
  "budgetMin": 10.00,
  "budgetMax": 20.00,
  "waitingToleranceMinutes": 10
}
```

校验规则：

1. `userType` 必须属于枚举值。
2. `budgetMin`、`budgetMax` 必须大于等于 0。
3. `budgetMin` 必须小于等于 `budgetMax`。
4. `waitingToleranceMinutes` 取值范围为 0-120。

### 5.2 场景配置 `ScenarioConfig`

```json
{
  "scenarioId": "weekday-lunch-peak",
  "name": "工作日午餐高峰",
  "mealPeriod": "LUNCH",
  "dayType": "WEEKDAY",
  "crowdLevel": "BUSY",
  "weatherFactor": 1.0,
  "eventFactor": 1.2,
  "closedWindowIds": [3],
  "virtualUserCount": 300,
  "durationMinutes": 60,
  "stepMinutes": 5,
  "randomSeed": 20260425
}
```

校验规则：

1. `mealPeriod`、`dayType`、`crowdLevel` 必须属于枚举值。
2. `weatherFactor`、`eventFactor` 必须大于 0，建议范围为 0.5-2.0。
3. `virtualUserCount` 取值范围为 1-5000。
4. `durationMinutes` 取值范围为 5-180。
5. `stepMinutes` 取值范围为 1-30，且 `durationMinutes` 应能被 `stepMinutes` 整除。
6. `randomSeed` 可选；传入同一 seed 时，A 应尽量保证结果可复现。
7. `timePoints` 数量应为 `durationMinutes / stepMinutes + 1`，必须包含 `minute = 0` 的初始状态。

### 5.3 餐厅参数 `RestaurantParameter`

```json
{
  "restaurantId": 1,
  "name": "一食堂",
  "location": "东区",
  "capacity": 500,
  "baseAttraction": 0.85,
  "status": "OPEN"
}
```

校验规则：

1. `capacity` 必须大于 0。
2. `baseAttraction` 取值范围为 0-1。
3. `status` 必须为 `OPEN` 或 `CLOSED`。

### 5.4 窗口参数 `WindowParameter`

```json
{
  "windowId": 1,
  "restaurantId": 1,
  "name": "川味窗口",
  "serviceRatePerMinute": 1.5,
  "status": "OPEN"
}
```

校验规则：

1. `serviceRatePerMinute` 必须大于 0。
2. `restaurantId` 必须对应存在的餐厅。
3. `status` 必须为 `OPEN` 或 `CLOSED`。

### 5.5 菜品参数 `DishParameter`

```json
{
  "dishId": 12,
  "windowId": 1,
  "name": "麻辣香锅",
  "price": 18.00,
  "prepTimeMinutes": 4,
  "popularity": 0.9,
  "tags": ["偏辣", "米饭"]
}
```

校验规则：

1. `price` 必须大于等于 0。
2. `prepTimeMinutes` 必须大于 0。
3. `popularity` 取值范围为 0-1。
4. `windowId` 必须对应存在的窗口。

### 5.6 仿真输入 `SimulationRunRequest`

```json
{
  "profile": {
    "userType": "STUDENT",
    "tasteTags": ["偏辣", "米饭"],
    "budgetMin": 10.00,
    "budgetMax": 20.00,
    "waitingToleranceMinutes": 10
  },
  "scenario": {
    "mealPeriod": "LUNCH",
    "dayType": "WEEKDAY",
    "crowdLevel": "BUSY",
    "weatherFactor": 1.0,
    "eventFactor": 1.2,
    "closedWindowIds": [3],
    "virtualUserCount": 300,
    "durationMinutes": 60,
    "stepMinutes": 5,

  }
}
```

说明：

1. MVP 阶段请求中必须包含 `profile` 和 `scenario`。
2. 餐厅、窗口、菜品参数默认使用系统预置数据。
3. 如需临时覆盖参数，后续可扩展 `parameterOverrides` 字段。

### 5.7 仿真时间点 `SimulationTimePoint`

```json
{
  "minute": 10,
  "restaurants": [
    {
      "restaurantId": 1,
      "name": "一食堂",
      "currentCount": 180,
      "capacity": 500,
      "crowdLevel": "NORMAL",
      "windows": [
        {
          "windowId": 1,
          "name": "川味窗口",
          "queueLength": 18,
          "servingCount": 2,
          "waitMinutes": 12,
          "crowdLevel": "BUSY",
          "status": "OPEN"
        }
      ]
    }
  ]
}
```

说明：

1. `minute` 表示仿真运行后的第几分钟。
2. `restaurants` 必须包含本次仿真范围内所有餐厅。
3. `windows` 必须包含对应餐厅下所有参与展示的窗口。
4. 已关闭窗口也应返回，`status` 为 `CLOSED`，队列和等待时间可为 0。


## 8. 成员 C 前端调用顺序

### 8.1 页面初始化

C 打开仿真控制台时调用：

```text
GET /api/v1/presets/user-profiles
GET /api/v1/presets/scenarios
GET /api/v1/parameters/restaurants
GET /api/v1/parameters/windows
GET /api/v1/parameters/dishes
```

用途：

1. 渲染用户画像模板。
2. 渲染场景模板。
3. 准备餐厅、窗口、菜品基础信息。

### 8.2 运行一次仿真

C 点击“运行仿真”时调用：

```text
POST /api/v1/simulations/run
```

成功后 C 应保存：

```text
runId
timePoints
metrics
```

然后 C 可以直接用 `timePoints` 播放动态图表。

### 8.3 生成推荐

C 拿到 `runId` 后调用：

```text
POST /api/v1/recommendations/generate
```

成功后 C 展示：

1. 推荐餐厅。
2. 推荐窗口。
3. 推荐菜品。
4. 分流建议。

### 8.4 动态展示全体排队状况

C 使用 `timePoints` 按 `minute` 播放。

每次切换时间点时，页面刷新：

1. 全体餐厅当前人数。
2. 全体窗口排队长度。
3. 各窗口预计等待时间。
4. 拥挤度颜色。
5. 当前时间点推荐结果。

MVP 阶段不要求后端实时推送，前端可用一次性返回的 `timePoints` 做时间轴播放。

### 8.5 场景对比

C 先后运行两次仿真，获得：

```text
baseRunId
compareRunId
```

然后调用：

```text
POST /api/v1/strategies/compare
```

前端展示平均等待时间、最大排队长度、拥挤窗口数和推荐结果变化。

## 10. 接口联调约定

### 10.1 开发优先级

第一优先级：

1. `POST /api/v1/simulations/run`
2. `POST /api/v1/recommendations/generate`
3. C 页面接入上述两个接口

第二优先级：

1. `GET /api/v1/presets/user-profiles`
2. `GET /api/v1/presets/scenarios`
3. `GET /api/v1/parameters/restaurants`
4. `GET /api/v1/parameters/windows`
5. `GET /api/v1/parameters/dishes`

第三优先级：

1. `GET /api/v1/simulations/{runId}`
2. `GET /api/v1/simulations/{runId}/timeline`
3. `GET /api/v1/simulations/{runId}/metrics`
4. `GET /api/v1/recommendations/runs/{runId}`
5. `POST /api/v1/recommendations/diversion`
6. `POST /api/v1/strategies/compare`

### 10.2 Mock 数据约定

为避免 C 等待 A、B 完成后才能开发，A 和 B 应尽早提供固定 Mock JSON。

最低要求：

1. 至少 2 个餐厅。
2. 每个餐厅至少 2 个窗口。
3. 至少 6 个菜品。
4. 至少 5 个时间点。
5. 至少 1 组推荐结果。
6. 至少 1 组场景对比结果。

### 10.3 命名约定

1. ID 字段统一为数字，例如 `runId`、`restaurantId`、`windowId`、`dishId`。
2. 枚举值统一使用大写英文。
3. 前端展示中文名称时使用 `name` 字段。
4. 推荐理由由 B 直接返回中文文本，C 不负责拼接复杂推荐理由。

### 10.4 结果一致性约定

1. A 返回的 `runId` 是一次仿真的唯一标识。
2. B 所有推荐结果必须关联 `runId`。
3. C 展示推荐结果时，必须保证推荐结果和当前展示的仿真结果属于同一个 `runId`。
4. 如果 C 切换了时间点，推荐结果可以重新调用 B 接口，也可以先使用默认推荐结果。

### 10.5 单接口验收标准

每个接口完成时，至少满足以下条件：

1. 可以用浏览器、Postman 或 curl 访问。
2. 成功和失败都使用统一响应结构。
3. 请求字段缺失或类型错误时返回明确错误码和错误信息。
4. 返回字段名称、类型、枚举值和本文档一致。
5. Mock 数据不能只返回空数组，必须能支撑 C 做页面展示。
6. 同一个 `runId` 查询到的仿真结果、推荐结果、对比结果必须前后一致。

## 11. 最小可演示闭环

最终演示至少需要跑通以下闭环：

1. C 加载用户画像和场景预设。
2. C 选择用户画像和场景参数。
3. C 调用 A 的 `POST /api/v1/simulations/run`。
4. A 返回 `runId`、`timePoints` 和 `metrics`。
5. C 用 `timePoints` 展示全体餐厅和窗口排队状态。
6. C 调用 B 的 `POST /api/v1/recommendations/generate`。
7. B 返回推荐餐厅、推荐窗口、推荐菜品和分流建议。
8. C 展示推荐结果和推荐理由。
9. C 修改场景再次运行仿真。
10. C 调用 B 的 `POST /api/v1/strategies/compare` 展示对比结论。
