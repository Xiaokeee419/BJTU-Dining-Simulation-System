# 北京交通大学就餐仿真系统接口规范

版本：v0.1  
日期：2026-04-19  
适用范围：前端、后端、数据库、仿真与推荐模块联调

## 1. 项目规模评估

本项目不是单纯的页面展示项目，而是一个小型到中型的 B/S 全栈系统。系统包含 Vue3 前端、Spring Boot 后端、MySQL 持久化、Redis 缓存、JWT 鉴权、WebSocket 实时推送，以及人流仿真和推荐算法逻辑。

按课程项目标准，规模评估如下：

| 维度 | 评估 |
| --- | --- |
| 功能模块 | 5 个核心业务模块：用户中心、餐厅、菜品、流量统计、个性化推荐 |
| 页面数量 | 约 6 个主要页面：登录注册、首页推荐、餐厅详情、菜品详情、个性化推荐、用户中心 |
| 数据实体 | 约 10 个核心实体：User、Wallet、UserPreference、Restaurant、Window、Dish、FlowSnapshot、FlowRule、RecommendationRecord、Rating、Transaction |
| 接口数量 | MVP 阶段约 25-35 个 REST 接口，外加 1 个 WebSocket 实时通道 |
| 技术复杂度 | 中等：普通 CRUD 较多，但 JWT、余额事务、仿真数据、推荐计算、WebSocket 会增加集成难度 |
| 团队规模匹配度 | 3 人可以完成，但需要较早约定接口和数据结构，避免最后集中联调 |
| 建议开发周期 | MVP 约 2-3 周；包含较完整页面、评分、推荐、实时推送和测试数据初始化时约 4-6 周 |

建议先完成 MVP：

1. 用户可以注册、登录、查看余额、模拟充值和消费。
2. 首页可以展示餐厅推荐和当前拥挤度。
3. 餐厅详情可以展示窗口、菜品、等待时间。
4. 菜品详情可以查看菜品信息并提交评分。
5. 推荐模块先使用可解释的规则算法，不必一开始追求复杂模型。
6. WebSocket 先推送餐厅和窗口实时人数，图表趋势可以后续补强。

## 3. 全局接口约定

### 3.1 基础路径

所有 REST 接口统一以 `/api/v1` 为前缀。

示例：

```text
GET /api/v1/restaurants
POST /api/v1/auth/login
```

WebSocket 地址：

```text
ws://localhost:8080/ws/flow
```

### 3.2 数据格式

1. 请求和响应统一使用 JSON。
2. HTTP Header 使用 `Content-Type: application/json`。
3. JSON 字段统一使用 `camelCase`，数据库字段可以继续使用 `snake_case`。
4. 时间统一使用 ISO-8601 字符串，例如 `2026-04-19T12:30:00+08:00`。
5. 金额使用数字类型，保留两位小数，例如 `12.50`。
6. ID 使用数字类型，前后端不要自行拼接业务含义。

### 3.3 鉴权约定

除注册、登录、餐厅列表、餐厅详情、菜品列表、菜品详情、实时人流查询等公开接口外，其余接口默认需要 JWT。

Header 格式：

```http
Authorization: Bearer <token>
```

后端从 JWT 中解析 `userId`，前端不要在需要鉴权的接口中额外传入当前登录用户 ID，避免越权。

### 3.4 统一响应结构

成功响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "traceId": "9bdb3c3c6c7b4f41",
  "timestamp": "2026-04-19T12:30:00+08:00"
}
```

失败响应：

```json
{
  "code": 40001,
  "message": "参数校验失败",
  "data": {
    "field": "password",
    "reason": "密码长度不能少于 6 位"
  },
  "traceId": "9bdb3c3c6c7b4f41",
  "timestamp": "2026-04-19T12:30:00+08:00"
}
```

### 3.5 分页结构

分页参数统一使用：

```text
page=1&pageSize=10
```

分页响应：

```json
{
  "list": [],
  "total": 100,
  "page": 1,
  "pageSize": 10
}
```

### 3.6 HTTP 状态码

| 状态码 | 含义 |
| --- | --- |
| 200 | 请求成功 |
| 201 | 创建成功 |
| 400 | 请求参数错误 |
| 401 | 未登录或 Token 无效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 资源冲突，例如用户名重复或余额不足 |
| 500 | 服务端异常 |

### 3.7 业务错误码

| code | 含义 |
| --- | --- |
| 0 | 成功 |
| 40000 | 请求格式错误 |
| 40001 | 参数校验失败 |
| 40100 | 未登录 |
| 40101 | Token 过期 |
| 40300 | 无权限 |
| 40400 | 资源不存在 |
| 40900 | 资源状态冲突 |
| 40901 | 余额不足 |
| 50000 | 服务端异常 |
| 50300 | 仿真或推荐服务暂不可用 |

## 5. 成员 A：用户与账户中心接口

### 5.1 注册

```http
POST /api/v1/auth/register
```

请求：

```json
{
  "username": "student001",
  "password": "123456",
  "nickname": "小明"
}
```

响应 `data`：

```json
{
  "userId": 1,
  "username": "student001",

}
```

1. `username` 全局唯一。

## 9. 模块之间的数据交流规范

### 9.1 餐厅推荐

数据来源：

| 来源模块 | 提供数据 | 接口 |
| --- | --- | --- |
| 用户模块 | 用户 ID、口味偏好、历史评分、浏览和消费记录 | `/users/me/preferences`、`/ratings/my`、`/transactions` |
| 人流模块 | 当前人数、拥挤度、排队长度、预计等待时间 | `/flows/restaurants/current` |
| 餐厅模块 | 餐厅基础信息、营业状态、容量 | `/restaurants/**` |

推荐模块输出：

```http
GET /api/v1/recommendations/restaurants
```

前端展示时需要同时展示推荐理由、当前人数、拥挤等级和预计等待时间。

### 9.2 菜品推荐

数据来源：

| 来源模块 | 提供数据 | 接口 |
| --- | --- | --- |
| 用户模块 | 用户偏好、历史评分、消费记录 | `/users/me/preferences`、`/ratings/my`、`/transactions` |
| 菜品模块 | 菜品标签、价格、评分、库存 | `/dishes/**` |
| 餐厅模块 | 餐厅、窗口、营业状态 | `/restaurants/**`、`/windows/**` |
| 人流模块 | 当前窗口排队长度、预计等待时间 | `/flows/windows/{windowId}/current` |

推荐模块输出：

```http
GET /api/v1/recommendations/dishes
```

推荐计算建议先使用规则：

```text
推荐分 = 偏好匹配分 * 0.45 + 菜品评分分 * 0.30 + 价格匹配分 * 0.10 - 等待时间惩罚 * 0.15
```

具体权重可由成员 C 调整，但响应字段必须保持不变。

### 9.3 点餐与消费

交互链路：

```text
菜品详情页 -> POST /orders -> 用户账户中心扣款 -> 生成交易记录 -> 返回订单和最新余额
```

接口归属：

| 动作 | 负责成员 | 接口 |
| --- | --- | --- |
| 展示菜品和价格 | 成员 B | `GET /dishes/{dishId}` |
| 提交点餐请求 | 成员 A | `POST /orders` |
| 扣减余额和生成流水 | 成员 A | `/wallet/**`、`/transactions/**` |
| 展示消费后的评分入口 | 前端页面协作 | `POST /ratings` |

注意：

1. 前端不要直接调用扣余额接口，统一通过 `POST /orders` 完成。
2. 后端必须以后端查询到的菜品价格为准，不能信任前端传来的价格。
3. 余额扣减、订单创建、交易记录创建必须放在同一事务中。

### 9.4 评分系统

交互链路：

```text
餐厅详情页/菜品详情页/消费后弹窗 -> POST /ratings -> 更新目标对象平均分 -> 推荐模块后续读取评分
```

接口归属：

| 动作 | 负责成员 | 接口 |
| --- | --- | --- |
| 提交评分记录 | 成员 A | `POST /ratings` |
| 查询我的评分 | 成员 A | `GET /ratings/my` |
| 查询菜品或窗口评分列表 | 成员 A 提供接口，成员 B 页面调用 | `GET /ratings/targets/{targetType}/{targetId}` |
| 展示综合评分 | 成员 B | `GET /dishes/{dishId}`、`GET /windows/{windowId}` |
| 推荐计算使用评分 | 成员 C | 内部读取评分数据或调用评分查询接口 |

建议：

1. 评分记录以成员 A 的 `Rating` 表为准。
2. 菜品、窗口、餐厅的平均分可以由成员 B 查询时聚合，也可以由成员 A 在评分提交后回写统计字段。
3. MVP 阶段推荐使用“提交评分后回写平均分”的方式，前端查询更简单。

## 10. 前后端联调约定

### 10.1 前端调用顺序

登录后：

```text
POST /auth/login
GET /users/me
GET /wallet
GET /recommendations/restaurants
GET /flows/restaurants/current
连接 /ws/flow
```

进入餐厅详情页：

```text
GET /restaurants/{restaurantId}
GET /restaurants/{restaurantId}/windows
GET /flows/restaurants/{restaurantId}/current
```

进入菜品详情页：

```text
GET /dishes/{dishId}
GET /ratings/targets/DISH/{dishId}
GET /flows/windows/{windowId}/current
```

点餐后：

```text
POST /orders
GET /wallet
可选：POST /ratings
```

### 10.2 Mock 数据约定

在接口尚未完成时，各成员可以先按本文的 JSON 结构提供 Mock。Mock 必须满足：

1. 字段名、字段类型与本文保持一致。
2. 枚举值只使用本文列出的值。
3. 时间字段使用 ISO-8601 字符串。
4. 金额字段使用数字，不使用字符串。

### 10.3 数据库和接口命名关系

| 数据库字段 | JSON 字段 |
| --- | --- |
| user_id | userId |
| restaurant_id | restaurantId |
| window_id | windowId |
| dish_id | dishId |
| current_count | currentCount |
| queue_length | queueLength |
| create_time | createTime |
| update_time | updateTime |

### 10.4 兼容性规则

1. 已经被前端使用的响应字段不要随意删除或改名。
2. 新增字段可以直接追加，前端应忽略未知字段。
3. 枚举值新增前需要在小组内同步。
4. 接口路径确定后不要频繁修改；确需修改时保留旧路径一段时间或统一一次性替换。
