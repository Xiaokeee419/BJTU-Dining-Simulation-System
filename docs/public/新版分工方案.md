# 新版分工方案

版本：v0.2

日期：2026/4/25

适用范围：就餐仿真系统后续开发、Issue 拆分、PR 归属和联调验收

---

## 1. 分工调整背景

在开发阶段讨论后，项目定位从偏“食堂信息管理系统”调整为“校园就餐场景仿真与决策支持系统”。

因此，后续分工不再围绕注册登录、用户中心、钱包、订单等管理流程展开，而是围绕仿真系统的核心链路展开：

```text
仿真数据生成
-> 分流与推荐决策
-> 前端控制台与可视化展示
```

本方案目标是保证三名成员任务量尽量相当，同时保持模块边界清晰，减少联调时的职责重叠。

---

## 2. 总体分工

| 成员 | 负责模块 | 核心定位 |
| --- | --- | --- |
| 成员 A | 仿真数据生成与效果评估模块 | 负责生成仿真所需数据，并输出排队、等待、拥挤度和评估指标 |
| 成员 B | 分流策略与推荐算法模块 | 负责基于 A 的仿真结果做分流决策、餐厅推荐和菜品推荐 |
| 成员 C | 仿真控制台与可视化展示模块 | 负责网页端参数配置、动态展示、图表和推荐结果展示 |

三者关系：

```text
成员 A 生成仿真数据和评估指标
        ↓
成员 B 基于仿真结果进行分流和推荐
        ↓
成员 C 展示仿真过程、排队状态和推荐结果
```

---

## 3. 成员 A：仿真数据生成与效果评估模块

### 3.1 模块定位

成员 A 负责“仿真结果怎么来”。

该模块不是简单造静态测试数据，而是根据用户画像、餐厅窗口参数、菜品参数和场景配置，生成虚拟就餐者及其在仿真系统中的人流、排队和等待结果。

### 3.2 主要任务

1. 定义仿真输入数据结构。
2. 生成虚拟就餐者数据。
3. 根据场景参数生成不同时间片的人流数据。
4. 计算各餐厅和窗口的排队长度。
5. 计算各窗口预计等待时间。
6. 判断餐厅和窗口拥挤度等级。
7. 输出完整的仿真时间序列数据。
8. 计算效果评估指标，例如平均等待时间、最大排队长度、拥挤窗口数量。

### 3.3 输入数据

成员 A 需要接收或读取以下输入：

| 数据 | 说明 |
| --- | --- |
| 用户画像 | 用户类型、口味偏好、预算范围、等待容忍度 |
| 餐厅参数 | 餐厅容量、位置、基础吸引力、营业状态 |
| 窗口参数 | 窗口服务速率、窗口状态、所属餐厅 |
| 菜品参数 | 菜品价格、出餐时间、热门度、标签 |
| 场景参数 | 早餐/午餐/晚餐、工作日/周末、拥挤等级、特殊事件、仿真人数 |

### 3.4 输出数据

成员 A 输出给成员 B 和成员 C 的核心数据包括：

| 输出数据 | 说明 | 主要使用方 |
| --- | --- | --- |
| `VirtualDiner` | 虚拟就餐者数据 | B、C |
| `FlowSnapshot` | 每个时间片的餐厅/窗口人流快照 | B、C |
| `QueueState` | 各窗口排队长度、服务中人数、等待时间 | B、C |
| `CrowdLevel` | 餐厅/窗口拥挤度等级 | B、C |
| `EvaluationMetrics` | 平均等待时间、最大排队长度等评估指标 | B、C |

### 3.5 不负责内容

成员 A 不负责：

1. 推荐排序算法。
2. 分流策略设计。
3. 前端页面开发。
4. 图表展示。
5. 用户注册、钱包、订单等非仿真核心流程。

### 3.6 验收标准

成员 A 的模块应满足：

1. 可以根据不同场景参数生成不同仿真结果。
2. 可以输出所有餐厅和窗口的排队状态。
3. 可以输出等待时间和拥挤度。
4. 可以生成趋势数据，供前端动态播放。
5. 可以提供评估指标，供 B 进行策略比较和推荐计算。

## 4. 成员 B：分流策略与推荐算法模块

### 4.1 模块定位

成员 B 负责“基于仿真结果怎么决策”。

该模块不负责从零生成原始人流数据，也不负责维护完整仿真时间片。成员 B 主要基于成员 A 生成的人流、排队、等待和拥挤度结果，计算分流策略、推荐排序和推荐理由。

### 4.2 主要任务

1. 设计人流分流策略。
2. 基于等待时间和拥挤度判断是否需要引导用户分流。
3. 计算餐厅推荐分数。
4. 计算窗口推荐分数。
5. 计算菜品推荐分数。
6. 生成推荐理由。
7. 对比分流前后或不同场景下的策略效果。
8. 输出推荐结果和策略比较结果。

### 4.3 输入数据

成员 B 主要使用成员 A 的输出数据：

| 数据 | 说明 |
| --- | --- |
| `FlowSnapshot` | 当前或多个时间片的人流快照 |
| `QueueState` | 各窗口排队长度和等待时间 |
| `CrowdLevel` | 各餐厅/窗口拥挤度 |
| `EvaluationMetrics` | 仿真效果评估指标 |
| 用户画像 | 预算、口味、等待容忍度 |
| 菜品参数 | 价格、标签、热门度、出餐时间 |

### 4.4 输出数据

成员 B 输出给成员 C 的核心数据包括：

| 输出数据 | 说明 |
| --- | --- |
| `RestaurantRecommendation` | 餐厅推荐结果 |
| `WindowRecommendation` | 窗口推荐结果 |
| `DishRecommendation` | 菜品推荐结果 |
| `DiversionSuggestion` | 分流建议 |
| `StrategyComparison` | 不同策略或场景的比较结果 |

### 4.5 推荐逻辑建议

推荐分数可以采用规则加权方式，不需要一开始使用复杂机器学习模型。

示例评分因素：

| 因素 | 影响 |
| --- | --- |
| 口味匹配度 | 匹配越高，推荐分越高 |
| 预算匹配度 | 价格在预算范围内，推荐分越高 |
| 等待时间 | 等待越久，推荐分越低 |
| 拥挤度 | 越拥挤，推荐分越低 |
| 菜品热门度 | 热门度越高，推荐分越高 |
| 出餐时间 | 出餐越慢，推荐分越低 |

推荐理由示例：

```text
推荐二食堂川味窗口：符合偏辣口味，预算匹配，预计等待 6 分钟，低于用户等待容忍度 10 分钟，当前拥挤度较低。
```

### 4.6 不负责内容

成员 B 不负责：

1. 原始虚拟就餐者生成。
2. 原始排队时间序列生成。
3. 前端页面开发。
4. 图表绘制。
5. 数据库基础参数维护。

### 4.7 验收标准

成员 B 的模块应满足：

1. 可以基于 A 的仿真结果生成餐厅、窗口和菜品推荐。
2. 推荐结果可以随场景变化而变化。
3. 推荐理由具备可解释性。
4. 可以输出分流建议。
5. 可以比较不同场景或策略下的效果差异。

## 5. 成员 C：仿真控制台与可视化展示模块

### 5.1 模块定位

成员 C 负责“如何把仿真过程和结果展示给用户”。

该模块是系统演示入口，重点不是做传统管理后台，而是做一个图形化的仿真控制台，使老师和使用者能够直观看到参数输入、仿真过程、排队变化和推荐结果。

### 5.2 主要任务

1. 开发仿真控制台页面。
2. 提供用户画像配置界面。
3. 提供场景配置界面。
4. 提供运行仿真按钮。
5. 动态展示全体餐厅和窗口排队状况。
6. 展示等待时间、拥挤度和人流趋势图。
7. 展示餐厅、窗口和菜品推荐结果。
8. 展示场景对比结果。

### 5.3 输入数据

成员 C 主要接收：

| 数据 | 来源 |
| --- | --- |
| 用户画像配置项 | 页面输入或预置模板 |
| 场景配置项 | 页面输入或预置模板 |
| `FlowSnapshot` | 成员 A |
| `QueueState` | 成员 A |
| `EvaluationMetrics` | 成员 A |
| 推荐结果 | 成员 B |
| 分流建议 | 成员 B |

### 5.4 输出内容

成员 C 需要在页面上展示：

1. 仿真参数配置面板。
2. 全体餐厅当前人数。
3. 全体窗口排队长度。
4. 各窗口预计等待时间。
5. 拥挤度状态卡片。
6. 人流趋势图。
7. 推荐结果卡片。
8. 场景对比图表。

### 5.5 页面建议

MVP 阶段建议优先实现一个页面：

```text
仿真控制台页
```

页面可以划分为：

| 区域 | 内容 |
| --- | --- |
| 左侧 | 用户画像配置 |
| 中间 | 场景参数配置 |
| 右侧 | 推荐结果 |
| 下方 | 排队、等待时间和拥挤度图表 |

### 5.6 不负责内容

成员 C 不负责：

1. 人流生成算法。
2. 排队等待计算。
3. 推荐排序算法。
4. 后端数据持久化。

### 5.7 验收标准

成员 C 的模块应满足：

1. 可以在页面配置用户画像和场景参数。
2. 可以触发一次仿真运行。
3. 可以展示全体餐厅和窗口排队状态。
4. 可以用图表展示等待时间和拥挤度趋势。
5. 可以展示 B 输出的推荐结果和推荐理由。
6. 可以支持至少两种场景结果对比。

## 7. 推荐开发顺序

### 7.1 第一阶段：接口和假数据对齐

1. A 定义仿真结果 JSON 结构。
2. B 定义推荐结果 JSON 结构。
3. C 用假数据先搭页面和图表。

### 7.2 第二阶段：模块独立开发

1. A 完成人流数据生成和评估指标。
2. B 完成分流策略和推荐算法。
3. C 完成仿真控制台和动态图表。

### 7.3 第三阶段：联调与演示

1. C 接入 A 的仿真结果。
2. C 接入 B 的推荐结果。
3. 三人共同调试场景切换、结果变化和推荐解释。
4. 准备最终演示流程。

---

## 8. 最终演示主线

推荐演示流程如下：

1. 打开仿真控制台。
2. 配置用户画像，例如学生、偏辣、预算 10-20 元、最多等待 10 分钟。
3. 配置场景，例如工作日午餐高峰、某窗口关闭、仿真人数 300 人。
4. 运行仿真。
5. 展示全体餐厅和窗口排队状态。
6. 展示等待时间、拥挤度和趋势图。
7. 展示推荐餐厅、推荐窗口和推荐菜品。
8. 切换场景或策略，再次运行仿真。
9. 对比两次结果变化，说明分流和推荐的决策价值。

---

## 9. 一句话总结

成员 A 负责生成仿真数据和评估指标，成员 B 负责基于仿真结果做分流和推荐决策，成员 C 负责将仿真过程和推荐结果做成可交互、可视化的网页控制台。
