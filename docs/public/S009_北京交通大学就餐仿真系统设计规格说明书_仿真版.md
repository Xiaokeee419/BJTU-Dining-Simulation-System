## 1. 系统总体目标

### 1.1 建设背景

本系统原始方案中包含注册登录、用户中心、余额充值、订单消费、评分记录等功能，这些内容能够形成较完整的业务管理流程，但整体上更接近食堂信息管理或点餐管理系统。结合开发阶段讨论结果，项目目标需要重新聚焦于“仿真”本身，即通过参数配置、虚拟就餐者生成、人流变化计算和排队等待分析，模拟校园就餐场景下餐厅与窗口的运行状态。

因此，本系统修订为“校园就餐场景仿真与决策支持系统”。系统不再以真实用户注册、充值、下单为核心流程，而是以用户画像、餐厅窗口参数、菜品参数和就餐场景配置为输入，生成一批虚拟就餐个体，在仿真环境中模拟其到达、选择、排队和等待过程，并根据仿真结果给出餐厅和菜品推荐。

### 1.2 建设目标

本系统的建设目标如下：

1. 支持配置用户画像，包括用户类型、口味偏好、预算范围和等待容忍度。
2. 支持配置菜品特征参数，包括菜品价格、出餐时间、热门度和标签。
3. 支持配置餐厅与窗口参数，包括餐厅容量、窗口数量、窗口服务速率和营业状态。
4. 支持配置就餐场景，包括早餐、午餐、晚餐、工作日、周末、高峰、极端拥挤、窗口关闭、天气或活动影响等。
5. 根据输入参数生成虚拟就餐者，并模拟其在餐厅和窗口之间的选择、到达、排队与等待过程。
6. 输出仿真结果，包括餐厅人数、窗口排队长度、预计等待时间、拥挤度等级和趋势数据。
7. 基于用户画像和仿真结果生成餐厅推荐、窗口推荐和菜品推荐，并给出推荐理由。
8. 支持不同场景之间的策略比较，为校园就餐决策和食堂运营调整提供参考。

### 1.3 系统使用对象

本系统主要面向以下使用对象：

1. 项目演示人员：通过图形化页面配置参数、运行仿真并展示结果。
2. 普通就餐决策用户：通过用户画像和推荐结果了解更适合的餐厅、窗口或菜品选择。
3. 食堂运营分析人员：通过不同场景下的人流、排队和拥挤度结果，辅助评估窗口开放策略和服务能力。

### 1.4 系统定位

系统定位为仿真与决策支持工具，而不是传统的食堂管理系统。注册登录、余额充值、订单消费等功能不作为 MVP 阶段核心内容。用户模块在本系统中主要体现为“用户画像与行为偏好”，用于驱动仿真个体生成和推荐计算。

## 2. 系统开发架构

本系统采用 B/S（Browser/Server）架构，前端以 Web 页面形式提供仿真控制台，后端负责参数管理、仿真计算、推荐计算和结果持久化。

### 2.1 总体部署方式

1. 浏览器端提供仿真控制台页面，用于配置用户画像、场景参数并展示图表结果。
2. 后端服务提供 REST API，用于接收仿真输入、执行仿真计算、返回推荐结果。
3. 数据库用于存储餐厅、窗口、菜品、场景配置、仿真记录和推荐记录。
4. 缓存组件用于保存高频读取的仿真快照、推荐结果和运行状态。
5. WebSocket 可用于推送仿真过程中的动态人流变化，也可在 MVP 阶段以 REST 轮询方式替代。

### 2.2 架构说明

1. 前端页面负责参数配置、仿真触发、图表展示和推荐结果展示。
2. 后端服务负责接收仿真配置、组织仿真流程、计算排队等待和拥挤度，并调用推荐逻辑。
3. 仿真引擎负责根据场景参数生成虚拟就餐者，并模拟其选择餐厅、窗口和菜品的过程。
4. 推荐模块负责根据用户画像、菜品参数、窗口等待时间和拥挤度生成推荐结果。
5. MySQL 用于存储长期配置数据和仿真结果数据。
6. Redis 用于缓存当前运行中的仿真快照、热点餐厅窗口状态和推荐结果。
7. WebSocket 用于实时展示动态人流变化，提升演示时的可视化效果。

## 4. 系统架构

### 4.1 模块划分

本系统按照仿真实验的输入、计算和输出过程划分为 6 个核心模块：

1. 用户画像与行为偏好模块
2. 菜品特征参数模块
3. 餐厅与窗口场景参数模块
4. 场景配置模块
5. 人流生成与排队仿真模块
6. 仿真驱动的决策推荐模块

### 4.2 模块功能说明

| 模块名称 | 主要功能 | 作用 |
| --- | --- | --- |
| 用户画像与行为偏好模块 | 配置用户类型、口味偏好、预算范围、等待容忍度 | 为虚拟就餐者生成和推荐计算提供用户侧参数 |
| 菜品特征参数模块 | 配置菜品价格、出餐时间、热门度、标签、所属窗口 | 为用户选择、窗口服务时间和菜品推荐提供基础参数 |
| 餐厅与窗口场景参数模块 | 配置餐厅容量、窗口数量、窗口服务速率、营业状态 | 定义仿真环境的承载能力和服务能力 |
| 场景配置模块 | 配置时间段、日期类型、拥挤等级、窗口关闭、天气或活动影响 | 定义一次仿真实验的外部条件 |
| 人流生成与排队仿真模块 | 生成虚拟就餐者，模拟到达、选择、排队和等待，输出人流趋势 | 是系统核心计算模块，体现仿真系统特征 |
| 仿真驱动的决策推荐模块 | 根据用户画像、等待时间、拥挤度、菜品特征生成推荐 | 为用户提供餐厅、窗口和菜品选择建议 |

### 4.3 模块协同关系

模块协同说明如下：

1. 演示人员在前端仿真控制台选择用户画像并配置场景参数。
2. 用户画像模块向仿真模块提供用户类型、预算、口味和等待容忍度。
3. 菜品参数模块向仿真模块提供价格、标签、热门度和出餐时间。
4. 餐厅与窗口参数模块向仿真模块提供容量、服务速率、窗口状态和营业状态。
5. 场景配置模块向仿真模块提供时间段、日期类型、拥挤等级和特殊事件影响系数。
6. 人流生成与排队仿真模块根据上述输入生成虚拟就餐者，并计算各餐厅和窗口的状态。
7. 推荐模块读取仿真结果，结合用户画像输出餐厅、窗口和菜品推荐。
8. 前端页面展示仿真趋势图、拥挤度、等待时间和推荐理由。

### 4.4 模块之间的数据交换

| 来源模块 | 目标模块 | 交换数据 | 接口形式 |
| --- | --- | --- | --- |
| 用户画像与行为偏好模块 | 人流生成与排队仿真模块 | 用户类型、口味偏好、预算范围、等待容忍度 | REST/JSON |
| 菜品特征参数模块 | 人流生成与排队仿真模块 | 菜品价格、出餐时间、热门度、标签 | REST/JSON |
| 餐厅与窗口场景参数模块 | 人流生成与排队仿真模块 | 餐厅容量、窗口服务速率、窗口状态 | REST/JSON |
| 场景配置模块 | 人流生成与排队仿真模块 | 时间段、日期类型、拥挤等级、特殊事件系数 | REST/JSON |
| 人流生成与排队仿真模块 | 推荐模块 | 等待时间、排队长度、拥挤度、趋势数据 | REST/JSON、WebSocket |
| 推荐模块 | 前端页面 | 推荐餐厅、推荐窗口、推荐菜品、推荐理由 | REST/JSON |
| 人流生成与排队仿真模块 | 前端页面 | 人流趋势、排队长度、等待时间、拥挤度 | REST/JSON、WebSocket |

### 4.5 接口设计形式

系统模块之间主要采用以下接口形式：

RESTful API：用于实现用户画像配置、菜品参数查询、餐厅窗口参数查询、场景配置、仿真运行、结果查询和推荐结果查询。

WebSocket：用于在仿真运行过程中向前端推送餐厅人数、窗口排队长度、等待时间和拥挤度变化。MVP 阶段可以先使用 REST 接口返回完整仿真结果，WebSocket 作为增强功能。

JSON 数据格式：作为前后端数据交换的统一格式，用于封装仿真输入参数、仿真结果和推荐结果。

数据库访问接口：后端通过 MyBatis-Plus 访问 MySQL，完成参数配置、仿真记录和推荐结果的持久化。

### 4.6 核心业务流程

#### 仿真实验运行流程

1. 演示人员进入仿真控制台。
2. 选择或配置用户画像，例如学生、偏辣、预算 10-20 元、最多等待 10 分钟。
3. 配置就餐场景，例如工作日午餐高峰、某窗口关闭、活动导致人流增加 30%。
4. 系统读取餐厅、窗口和菜品参数。
5. 系统根据画像和场景生成一批虚拟就餐者。
6. 虚拟就餐者根据预算、口味、等待容忍度和菜品吸引力选择餐厅或窗口。
7. 仿真模块按时间片更新到达人数、排队长度、服务完成数量和等待时间。
8. 系统输出餐厅和窗口的人流趋势、拥挤度等级和预计等待时间。

#### 推荐决策流程

1. 推荐模块读取用户画像，包括预算、口味和等待容忍度。
2. 推荐模块读取仿真结果，包括窗口排队长度、预计等待时间和拥挤度。
3. 推荐模块读取菜品参数，包括价格、标签、热门度和出餐时间。
4. 系统按照“偏好匹配度 + 预算匹配度 + 等待时间惩罚 + 拥挤度惩罚”计算推荐分数。
5. 系统返回推荐餐厅、推荐窗口、推荐菜品和推荐理由。
6. 前端页面展示推荐结果，并标注等待时间和拥挤状态。

#### 场景对比流程

1. 演示人员选择基础场景，例如工作日午餐高峰。
2. 系统运行一次基础仿真并保存结果。
3. 演示人员修改场景参数，例如关闭一个窗口或增加窗口服务速率。
4. 系统再次运行仿真并保存结果。
5. 前端对比两次仿真的等待时间、排队长度、拥挤度和推荐结果变化。
6. 系统给出策略比较结论，用于说明不同运营策略对就餐体验的影响。

## 5. 软件原型

### 5.1 原型设计说明

系统前端采用 Web 页面形式，但不以登录注册、用户中心或点餐流程作为主要页面。MVP 阶段重点建设一个“就餐仿真控制台”页面，用于完成参数配置、仿真运行、结果可视化和推荐展示。

页面设计应突出“参数可配置、过程可观察、结果可对比”的特点，使老师和演示人员能够直观看到仿真系统的输入、计算和输出过程。

### 5.2 页面原型列表

| 页面名称 | 页面内容 | 用户可执行操作 |
| --- | --- | --- |
| 仿真控制台页 | 用户画像配置、场景配置、运行按钮、推荐结果、图表结果 | 配置参数、运行仿真、查看结果 |
| 参数预置页 | 餐厅参数、窗口参数、菜品参数、默认画像模板 | 查看或调整基础参数 |
| 仿真结果详情页 | 单次仿真的人流趋势、排队长度、等待时间、拥挤度 | 查看某次仿真实验详情 |
| 场景对比页 | 多个场景的等待时间、拥挤度和推荐结果对比 | 比较不同策略效果 |

### 5.3 主要页面示意

#### 5.3.1 仿真控制台页

仿真控制台页是系统 MVP 的核心页面。页面左侧为用户画像配置区，中间为场景配置区，右侧为推荐结果区，下方为图表展示区。

用户画像配置区包括用户类型、口味偏好、预算范围和等待容忍度。场景配置区包括时间段、日期类型、拥挤等级、特殊事件和仿真人数。推荐结果区展示推荐餐厅、推荐窗口、推荐菜品和推荐理由。图表展示区展示各餐厅人数、窗口排队长度、预计等待时间和拥挤度趋势。

#### 5.3.2 仿真结果详情页

仿真结果详情页用于展示一次仿真实验的完整结果。页面包含本次仿真的输入参数、虚拟就餐者数量、各餐厅分布、各窗口排队长度、平均等待时间和时间序列趋势图。

#### 5.3.3 场景对比页

场景对比页用于比较不同参数条件下的仿真结果。页面可以展示基础场景与调整场景在等待时间、拥挤度、推荐结果上的差异。例如比较“正常午高峰”和“关闭一个热门窗口”两种情况。

### 5.4 页面跳转关系

1. 用户进入系统后默认进入仿真控制台页。
2. 在仿真控制台页配置参数后，点击“运行仿真”生成结果。
3. 用户可从仿真控制台进入仿真结果详情页，查看更完整的趋势数据。
4. 用户可保存多个仿真结果，并进入场景对比页查看策略差异。
5. 参数预置页用于维护或调整餐厅、窗口、菜品和用户画像模板。

### 5.5 原型交互说明

1. 系统默认提供若干用户画像模板，例如普通学生、赶时间学生、预算敏感用户、偏辣用户。
2. 系统默认提供若干场景模板，例如工作日午餐高峰、周末晚餐、雨天拥挤、活动散场。
3. 用户可通过选择器、滑块和输入框调整参数。
4. 点击运行仿真后，页面刷新人流趋势图、排队长度图、等待时间图和推荐结果。
5. 用户修改场景参数后再次运行仿真，系统可展示两次结果差异。
6. 推荐结果必须给出可解释理由，例如“等待时间低于容忍度”“预算匹配”“口味标签匹配”“当前拥挤度较低”。

## 6. 数据模型

### 6.1 数据分类

系统数据可分为以下六类：

1. 用户画像参数数据
2. 菜品特征参数数据
3. 餐厅与窗口参数数据
4. 场景配置数据
5. 人流仿真结果数据
6. 推荐与策略比较数据

### 6.2 主要数据实体

| 实体名称 | 关键字段 | 数据说明 |
| --- | --- | --- |
| UserProfileTemplate | profile_id, name, user_type, taste_tags, budget_min, budget_max, waiting_tolerance_minutes | 存储预设用户画像模板 |
| DishParameter | dish_id, window_id, name, price, prep_time_minutes, popularity, tags | 存储菜品特征参数 |
| RestaurantParameter | restaurant_id, name, location, capacity, base_attraction, status | 存储餐厅容量和基础吸引力参数 |
| WindowParameter | window_id, restaurant_id, name, service_rate, status | 存储窗口服务能力和营业状态 |
| ScenarioConfig | scenario_id, meal_period, day_type, crowd_level, weather_factor, event_factor, virtual_user_count | 存储一次仿真实验的场景配置 |
| SimulationRun | run_id, scenario_id, profile_id, status, started_at, finished_at | 存储一次仿真运行记录 |
| VirtualDiner | diner_id, run_id, profile_id, arrival_time, selected_restaurant_id, selected_window_id | 存储虚拟就餐者生成结果，MVP 阶段可只在内存中生成 |
| FlowSnapshot | snapshot_id, run_id, restaurant_id, window_id, time_point, arrival_count, queue_length, wait_minutes, crowd_level | 存储某一时间片的人流和排队状态 |
| RecommendationResult | result_id, run_id, target_type, target_id, score, reason, rank | 存储推荐结果和推荐理由 |
| ScenarioComparison | comparison_id, base_run_id, compare_run_id, wait_delta, crowd_delta, conclusion | 存储不同场景之间的比较结果 |

### 6.3 数据产生位置

| 数据类型 | 数据来源 | 产生方式 |
| --- | --- | --- |
| 用户画像参数 | 仿真控制台、预置模板 | 用户选择或系统预置 |
| 餐厅与窗口参数 | 参数预置页、系统初始化数据 | 预置导入或手动调整 |
| 菜品参数 | 参数预置页、系统初始化数据 | 预置导入或手动调整 |
| 场景配置数据 | 仿真控制台 | 用户配置 |
| 虚拟就餐者数据 | 人流生成与排队仿真模块 | 后端算法生成 |
| 人流快照数据 | 人流生成与排队仿真模块 | 按时间片计算 |
| 推荐结果数据 | 仿真驱动的决策推荐模块 | 后端算法计算 |

### 6.4 数据流转过程

数据流转说明如下：

1. 系统初始化时加载餐厅、窗口和菜品基础参数。
2. 演示人员在仿真控制台选择用户画像并配置场景。
3. 后端根据用户画像和场景参数生成虚拟就餐者。
4. 仿真模块按时间片模拟虚拟就餐者到达、选择窗口、排队和完成服务。
5. 系统生成 FlowSnapshot，记录不同时间点的餐厅人数、窗口队列和等待时间。
6. 推荐模块读取仿真结果，并结合用户画像和菜品参数生成推荐列表。
7. 前端页面展示仿真图表、推荐结果和策略比较结论。
8. 仿真运行记录和结果可保存到数据库，用于后续查看和对比。

### 6.5 数据存储设计

| 数据类别 | 存储位置 | 存储说明 |
| --- | --- | --- |
| 用户画像、餐厅、窗口、菜品参数 | MySQL | 长期保存，支持复用和调整 |
| 场景配置和仿真运行记录 | MySQL | 保存每次仿真实验的输入与状态 |
| 人流快照和推荐结果 | MySQL、Redis | 重要结果落库，高频展示数据缓存 |
| 当前运行状态 | Redis | 保存运行中的仿真进度和临时结果 |
| 系统日志 | 文件日志或日志平台 | 用于故障排查和运行分析 |

### 6.6 数据安全设计

1. MVP 阶段不以真实用户注册为核心，系统尽量减少个人敏感信息存储。
2. 用户画像是仿真参数，不等同于真实用户账号。
3. 仿真参数和结果数据需要保证格式合法，防止异常参数导致计算错误。
4. 若后续加入登录功能，用户密码必须加密存储。
5. 参数修改和仿真运行应记录日志，便于复现实验结果。
6. 场景配置和推荐结果应与仿真运行记录关联，保证结果可追溯。

## 7. MVP 范围与验收指标

### 7.1 MVP 功能范围

MVP 阶段优先完成以下内容：

1. 一个 Web 仿真控制台页面。
2. 用户画像配置，包括用户类型、预算、口味和等待容忍度。
3. 场景配置，包括时间段、日期类型、拥挤等级、特殊事件和仿真人数。
4. 餐厅、窗口、菜品的基础参数预置。
5. 一次仿真运行接口，根据输入生成虚拟就餐者和排队结果。
6. 仿真结果展示，包括人数、队列、等待时间和拥挤度。
7. 推荐结果展示，包括餐厅推荐、窗口推荐、菜品推荐和推荐理由。
8. 至少支持两种场景对比，例如正常午高峰与窗口关闭场景。

### 7.2 非 MVP 核心内容

以下内容不作为 MVP 阶段核心展示内容：

1. 用户注册与登录。
2. 余额充值与钱包管理。
3. 真实点餐订单流程。
4. 复杂评论社区功能。
5. 完整后台管理系统。

已有相关代码可作为后续扩展能力保留，但不应作为本系统演示主线。

### 7.3 验收指标

系统验收时应能够完成以下演示：

1. 配置一个用户画像，例如预算 10-20 元、偏辣、最多等待 10 分钟的学生。
2. 配置一个场景，例如工作日午餐高峰、某窗口关闭、仿真人数 300 人。
3. 点击运行仿真后，系统生成餐厅和窗口的人流结果。
4. 页面展示排队长度、预计等待时间、拥挤度和趋势图。
5. 系统给出餐厅、窗口或菜品推荐，并说明推荐理由。
6. 调整场景参数后再次运行仿真，结果和推荐能够发生合理变化。

### 7.4 项目核心表述

本系统通过配置用户画像、菜品参数、餐厅窗口参数和就餐场景，批量生成虚拟就餐个体，在校园餐厅环境中进行人流与排队仿真，并基于仿真结果提供餐厅、窗口和菜品推荐决策。
