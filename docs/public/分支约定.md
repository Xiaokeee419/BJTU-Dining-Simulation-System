# 分支约定

本文档用于约定三人协作时的 Git 分支和提交流程。

当前开发分工以 `新版分工方案.md` 和 `接口规范.md` 为准：项目主线是“校园就餐场景仿真与决策支持系统”，不再按注册登录、钱包、订单等旧业务模块拆分。

## 1. 固定分支

| 分支 | 用途 | 规则 |
| --- | --- | --- |
| `main` | 稳定交付分支 | 只放可演示、可提交的版本，禁止直接开发 |
| `dev` | 日常集成分支 | 三个人的功能分支完成后统一合并到这里 |

## 2. 功能分支

每开发一个小功能，从 `dev` 新建一个功能分支。

命名格式：

```text
feature/成员-功能名
```

示例：

```text
feature/a-simulation-run
feature/a-parameter-preset
feature/b-recommendation-api
feature/b-strategy-compare
feature/c-simulation-console
feature/c-result-charts
```

成员对应关系：

| 成员 | 负责模块 | 分支前缀 |
| --- | --- | --- |
| 成员 A | 仿真数据生成与效果评估 | `feature/a-*` |
| 成员 B | 分流策略与推荐算法 | `feature/b-*` |
| 成员 C | 仿真控制台与可视化展示 | `feature/c-*` |

## 3. 开发流程

每天开始开发前先同步 `dev`：

```bash
git switch dev
git pull origin dev
```

新建功能分支：

```bash
git switch -c feature/b-recommendation-api
```

开发完成后提交：

```bash
git status
git add .
git commit -m "feat(recommendation): add recommendation api"
git push -u origin feature/b-recommendation-api
```

然后发起 Pull Request，目标分支选择 `dev`。

## 4. 合并规则

1. 不允许直接向 `main` 提交代码。
2. 不建议直接向 `dev` 提交代码，优先通过 Pull Request 合并。
3. 合并到 `dev` 前，至少保证项目能启动，自己负责的核心接口或页面可用。
4. 如果修改了公共接口，必须同步更新 `docs/public/接口规范.md`。
5. 如果修改了数据库结构，必须提交对应 SQL 文件或迁移说明。
6. 如果改到了别人负责的模块，需要提前在群里说明。

## 5. 提交信息

提交信息格式：

```text
类型(模块): 简短说明
```

示例：

```text
feat(simulation): add run api
feat(recommendation): add recommendation api
fix(strategy): handle missing metrics
docs(api): update recommendation interface
```

常用类型：

| 类型 | 含义 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修复问题 |
| `docs` | 文档 |
| `refactor` | 重构 |
| `test` | 测试 |
| `chore` | 配置、依赖、杂项 |

## 6. 发布版本

当 `dev` 已经稳定并通过联调后，再合并到 `main`：

```bash
git switch main
git pull origin main
git merge dev
git push origin main
```

重要节点可以打标签：

```bash
git tag v0.1-demo
git push origin v0.1-demo
```

推荐标签：

```text
v0.1-demo
v0.2-api
v0.3-websocket
v1.0-final
```
