# 分支约定

本文档用于约定三人协作时的 Git 分支和提交流程。

## 1. 固定分支

| 分支 | 用途 | 规则 |
| --- | --- | --- |
| `main` | 稳定交付分支 | 只放可演示、可提交的版本，禁止直接开发 |
| `dev` | 日常集成分支 | 三个人的功能分支完成后统一合并到这里 |

## 2. 功能分支

每开发一个小功能，从 `dev` 新建一个功能分支。

命名格式：

```text
feature/成员-功能名
```

示例：

```text
feature/a-login-jwt
feature/a-wallet
feature/b-restaurant-list
feature/b-dish-detail
feature/c-flow-simulation
feature/c-recommendation
```

成员对应关系：

| 成员 | 负责模块 | 分支前缀 |
| --- | --- | --- |
| 成员 A | 用户、登录、余额、订单、评分 | `feature/a-*` |
| 成员 B | 餐厅、窗口、菜品、首页基础展示 | `feature/b-*` |
| 成员 C | 人流仿真、推荐、WebSocket、图表数据 | `feature/c-*` |

## 3. 开发流程

每天开始开发前先同步 `dev`：

```bash
git switch dev
git pull origin dev
```

新建功能分支：

```bash
git switch -c feature/a-login-jwt
```

开发完成后提交：

```bash
git status
git add .
git commit -m "feat(auth): add login api"
git push -u origin feature/a-login-jwt
```

然后发起 Pull Request，目标分支选择 `dev`。

## 4. 合并规则

1. 不允许直接向 `main` 提交代码。
2. 不建议直接向 `dev` 提交代码，优先通过 Pull Request 合并。
3. 合并到 `dev` 前，至少保证项目能启动，自己负责的核心接口或页面可用。
4. 如果修改了公共接口，必须同步更新 `docs/接口规范.md`。
5. 如果修改了数据库结构，必须提交对应 SQL 文件或迁移说明。
6. 如果改到了别人负责的模块，需要提前在群里说明。

## 5. 提交信息

提交信息格式：

```text
类型(模块): 简短说明
```

示例：

```text
feat(auth): add login api
feat(dish): add dish detail page
fix(wallet): handle insufficient balance
docs(api): update rating interface
```

常用类型：

| 类型 | 含义 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修复问题 |
| `docs` | 文档 |
| `refactor` | 重构 |
| `test` | 测试 |
| `chore` | 配置、依赖、杂项 |

## 6. 发布版本

当 `dev` 已经稳定并通过联调后，再合并到 `main`：

```bash
git switch main
git pull origin main
git merge dev
git push origin main
```

重要节点可以打标签：

```bash
git tag v0.1-demo
git push origin v0.1-demo
```

推荐标签：

```text
v0.1-demo
v0.2-api
v0.3-websocket
v1.0-final
```
