## 1. 模块概述

本次个人模块对应项目中的成员 A，负责“仿真数据生成与效果评估模块”。该模块的任务不是做传统的登录、订单、钱包管理，而是围绕“北京交通大学就餐仿真系统”完成以下工作：

1. 整理虚拟学生、餐厅、窗口、菜品等基础数据。
2. 根据用户画像和场景参数生成虚拟就餐者。
3. 模拟不同时间片下的排队长度、等待时间和拥挤度。
4. 通过后端接口向成员 B 和成员 C 输出可联调的数据。

本模块当前采用“Python 数据整理脚本 + Spring Boot 后端接口”的实现方式。

## 3. 源代码文件说明

### 3.1 后端工程说明

#### `backend/pom.xml`

Maven 项目配置文件，定义了后端工程的基本信息和依赖。当前主要依赖包括：

1. `spring-boot-starter-web`：提供 REST 接口开发能力。
2. `spring-boot-starter-validation`：提供参数校验支持。
3. `spring-boot-starter-test`：为后续自动化测试预留测试依赖。

#### `backend/src/main/java/com/bjtu/dining/DiningSimulationApplication.java`

Spring Boot 启动类，用于启动成员 A 的后端服务。

#### `backend/src/main/java/com/bjtu/dining/common/ApiResponse.java`

统一响应结构封装类。所有接口均返回统一格式，便于前端和队友联调。

#### `backend/src/main/java/com/bjtu/dining/common/BadRequestException.java`

自定义参数错误异常类，用于表示请求参数不合法。

#### `backend/src/main/java/com/bjtu/dining/common/ResourceNotFoundException.java`

自定义资源不存在异常类，用于表示 `runId` 等资源查询失败。

#### `backend/src/main/java/com/bjtu/dining/common/GlobalExceptionHandler.java`

全局异常处理类，将参数错误、资源不存在和系统异常统一转换为标准接口响应。

#### `backend/src/main/java/com/bjtu/dining/config/WebConfig.java`

跨域配置类。用于允许前端本地开发端口访问后端接口，满足成员 C 联调需要。

### 3.2 成员 A 业务接口代码说明

#### `backend/src/main/java/com/bjtu/dining/taska/controller/PresetController.java`

预设接口控制器，负责提供用户画像预设和场景预设。

对应接口：

1. `GET /api/v1/presets/user-profiles`
2. `GET /api/v1/presets/scenarios`

#### `backend/src/main/java/com/bjtu/dining/taska/controller/ParameterController.java`

参数查询接口控制器，负责返回餐厅、窗口、菜品的基础参数。

对应接口：

1. `GET /api/v1/parameters/restaurants`
2. `GET /api/v1/parameters/windows`
3. `GET /api/v1/parameters/dishes`

#### `backend/src/main/java/com/bjtu/dining/taska/controller/SimulationController.java`

仿真运行接口控制器，负责接收仿真请求并返回仿真结果。

对应接口：

1. `POST /api/v1/simulations/run`
2. `GET /api/v1/simulations/{runId}`
3. `GET /api/v1/simulations/{runId}/timeline`
4. `GET /api/v1/simulations/{runId}/metrics`

#### `backend/src/main/java/com/bjtu/dining/taska/model/TaskADtos.java`

成员 A 模块的数据传输对象定义文件。主要包含：

1. 用户画像预设对象。
2. 场景预设对象。
3. 餐厅、窗口、菜品参数对象。
4. 仿真请求对象。
5. 仿真结果、时间片、指标对象。

该文件统一约束了 A 模块对外输出的数据结构。

#### `backend/src/main/java/com/bjtu/dining/taska/service/SeedDataService.java`

种子数据读取服务，负责从 `data/task-a/` 目录加载 CSV 文件并转换为内存对象。主要功能包括：

1. 读取虚拟学生、餐厅、窗口、菜品 CSV。
2. 提供餐厅参数查询、窗口参数查询、菜品参数查询。
3. 构建 `restaurantsById`、`windowsById`、`dishesByWindow` 等索引结构。

#### `backend/src/main/java/com/bjtu/dining/taska/service/SimulationService.java`

成员 A 的核心业务服务，负责完成仿真算法逻辑。主要工作包括：

1. 提供用户画像和场景预设。
2. 规范化请求参数并进行校验。
3. 从虚拟学生池中抽样生成本次仿真参与者。
4. 根据早餐、午餐、晚餐和拥挤等级调整到达时间。
5. 根据标签、预算、等待时间选择窗口和菜品。
6. 逐时间片更新各窗口队列状态。
7. 计算平均等待时间、最大排队长度、拥挤窗口数量等指标。
8. 将仿真结果缓存到内存中，供后续按 `runId` 查询。

#### `backend/src/main/resources/application.yml`

后端配置文件。当前主要用于配置应用端口和种子数据目录位置。

### 3.3 数据脚本与辅助文件说明

#### `scripts/prepare_task_a_seed_data.py`

成员 A 的数据整理脚本。该脚本读取两份 Excel 原始资料，并派生成仿真所需的关键 CSV 数据。主要作用包括：

1. 读取虚拟学生原始表。
2. 读取北交虚拟食堂和窗口原始表。
3. 生成用户类型、预算范围、等待容忍度等派生字段。
4. 生成早餐、午餐、晚餐的课程驱动到达时间。
5. 标准化标签，生成后续匹配使用的 `preference_tags` 和 `matching_tags`。

#### `scripts/run_task_a_simulation.py`

成员 A 的离线仿真原型脚本。该脚本直接读取 `data/task-a/*.csv`，输出一份与后端接口结构一致的示例结果 JSON，用于：

1. 在后端未完全启动前验证算法逻辑。
2. 生成前端或联调阶段可使用的 mock 数据。
3. 验证时间片、排队和指标计算是否合理。

#### `sql/schema/001_task_a_seed_schema.sql`

成员 A 的数据库表结构设计文件。用于定义虚拟学生、餐厅、窗口、菜品、到达规则和标签映射等数据表。

## 4. 数据文件说明

### `data/task-a/virtual_students.csv`

虚拟学生画像池，共 4169 条数据。用于仿真时抽样生成虚拟就餐者。

### `data/task-a/restaurants.csv`

餐厅基础数据，共 19 条记录。包括餐厅名称、位置、容量、基础吸引力、营业状态等字段。

### `data/task-a/windows.csv`

窗口基础数据，共 79 条记录。包括窗口名称、所属餐厅、服务速率、价格范围、热度和匹配标签等字段。

### `data/task-a/dishes.csv`

菜品基础数据，共 254 条记录。包括菜品名称、价格、出餐时间、热度和标签等字段。

### `data/task-a/arrival_rules.csv`

到达时间规则数据。用于表达早餐 8 点上课前、午餐 12 点下课后、晚餐 18 点 30 下课后的人流高峰特征。

### `data/task-a/tag_mappings.csv`

标签标准化映射文件。用于将原始口味、主食、饮食偏好标签统一到可计算的标准标签集合。

### `data/task-a/sample_simulation_result.json`

示例仿真结果文件。结构与后端 `POST /api/v1/simulations/run` 返回结果一致，可用于前端 mock 或结果展示说明。

### `data/task-a/README.md`

成员 A 种子数据说明文档，介绍了 CSV 文件来源、派生规则和文件对应关系。

## 6. 模块运行流程说明

成员 A 模块的基本运行流程如下：

```text
读取 Excel 原始资料
-> 运行 prepare_task_a_seed_data.py 生成 CSV
-> Spring Boot 后端启动并读取 data/task-a/*.csv
-> 前端或其他成员通过 /api/v1/presets、/api/v1/parameters、/api/v1/simulations 调用接口
-> 后端返回时间片、排队状态和评估指标
```
