## 3. 开发过程中遇到的问题及解决方法

### 3.1 项目定位前期不够清晰

开发初期，项目一度更偏向“食堂管理系统”，容易把工作重心放在注册、钱包、订单等传统管理模块上，这会导致成员 A 的职责边界不清晰。

解决方法：

1. 重新阅读规格说明书和分工文档。
2. 与队友统一项目定位，明确当前系统应以“就餐仿真与决策支持”为主。
3. 最终将成员 A 的任务收敛到“数据生成、排队仿真、评估指标”。

### 3.2 原始 Excel 数据不能直接用于程序读取

原始 Excel 更适合人工阅读，不适合直接参与程序计算。比如：

1. 字段命名不统一。
2. 标签表达方式不统一。
3. 缺少预算、等待容忍度、服务速率等仿真必需字段。

解决方法：

1. 编写 `prepare_task_a_seed_data.py` 进行数据清洗和派生。
2. 只保留程序真正需要的关键字段。
3. 对标签进行标准化，生成统一的匹配标签。

### 3.3 课程时间如何转化为人流高峰

系统不能只做静态数据展示，需要体现“仿真”特点。因此必须将上课、下课时间映射为早餐、午餐、晚餐高峰。

解决方法：

1. 将早餐高峰设置在 8 点上课前。
2. 将午餐高峰设置在 12 点下课后。
3. 将晚餐高峰设置在 18 点 30 下课后。
4. 在脚本中使用到达分钟字段和拥挤度缩放逻辑来体现人流集中程度。

### 3.4 接口结构需要兼顾 B 和 C 的联调需求

成员 A 的输出既要给成员 B 做推荐计算，又要给成员 C 做前端展示，因此接口字段不能只顾自己模块内部使用。

解决方法：

1. 按公共接口规范设计统一返回结构。
2. 将结果拆分为时间片数据和指标数据。
3. 提供预设接口、参数接口和仿真接口，便于不同成员调用。

### 3.5 本地环境和依赖问题

开发过程中遇到了 Maven 依赖、本地缓存和分支同步等问题。如果直接在公共分支上开发，容易影响队友。

解决方法：

1. 在自己的成员 A 分支上进行独立开发。
2. 使用项目内 `.m2` 本地仓库方式减少环境干扰。
3. 将公共 `dev` 与个人开发分支职责分开，避免生成数据长期占用公共分支。
